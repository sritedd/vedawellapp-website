## EXECUTIVE SUMMARY

The app has impressive breadth (50+ components, 20+ tables, AI, Stripe, offline) but several **security-critical and billing-critical paths are still MVP-style**. The biggest risks are:

1. **Billing/entitlement abuse** — Stripe checkout accepts any priceId, webhook trusts metadata blindly
2. **Missing DB hardening** — schemas v21-v26 are NOT applied, so RLS security hardening, phone verification, realtime are inactive in production
3. **Private data leakage** — Service worker caches authenticated pages and Supabase REST responses
4. **Phone verification is fake** — OTP is sent via email, not SMS, but marks phone as "verified"
5. **OTP logged in production** — Console.log leaks verification codes

**Top 3 priorities**: (1) Lock Stripe entitlement logic, (2) Disable private SW caching, (3) Apply pending migrations + move subscription mutations to service-role clients.

## P0 — FIX THIS WEEK (Critical)

### 🔴 BUG-01: Stripe Checkout Accepts Any priceId
**File**: `src/app/api/stripe/checkout/route.ts:18`
**Issue**: Client sends `priceId` and server uses it directly. A user could pass a cheaper/wrong Stripe price and still get upgraded to `guardian_pro`.
**Impact**: Revenue loss, billing bypass
**Fix**: Server-side allowlist of valid price IDs:
```typescript
const ALLOWED_PRICE_IDS = [process.env.STRIPE_PRO_MONTHLY_PRICE_ID!];
if (!ALLOWED_PRICE_IDS.includes(priceId)) {
    return NextResponse.json({ error: "Invalid plan" }, { status: 400 });
}
```

### 🔴 BUG-02: Stripe Webhook Blindly Trusts Metadata
**File**: `src/app/api/stripe/webhook/route.ts:88-107`
**Issue**: On `checkout.session.completed`, webhook reads `metadata.supabase_user_id` and upgrades to Pro without verifying the product/price matches Guardian Pro.
**Impact**: Entitlement abuse
**Fix**: Verify subscription product/price in webhook before upgrading tier.

### 🔴 SEC-01: Service Worker Caches Private Data
**File**: `public/sw.js:44-55, 62-84`
**Issue**: Caches Supabase REST API responses (including user data, projects, defects) and Guardian dashboard/project pages. Another user on the same device could see cached private data.
**Impact**: Data leakage between users, stale data bugs
**Fix**: Stop caching `/guardian/**` and `supabase.co/rest/*`. Bump `CACHE_NAME` to `vedawell-v3` to invalidate existing caches.

### 🔴 SEC-02: OTP Logged in Production
**File**: `src/app/api/guardian/phone-verify/route.ts:137`
**Issue**: `console.log(\`[Phone OTP] Code for ${phone}: ${otp}\`)` — OTP visible in server logs.
**Impact**: Anyone with log access can verify phones for any user
**Fix**: Remove the console.log immediately. Scrub existing logs.

### 🔴 SEC-03: Phone Verification Is Fake
**File**: `src/app/api/guardian/phone-verify/route.ts:134-173`
**Issue**: The OTP is sent via email, not SMS, but `phone_verified` is set to `true`. This proves email ownership, NOT phone ownership. Users can mark any unused phone number as "verified."
**Impact**: Phone uniqueness enforcement is meaningless
**Fix**: Either integrate actual SMS (Twilio) or rename to "contact verification" and don't set `phone_verified=true`.

### 🔴 SEC-04: Schemas v21-v26 Not Applied in Production
**Files**: `supabase/schema_v21_phone_verification.sql` through `schema_v26_security_hardening.sql`
**Issue**: Code references columns/tables from these schemas (phone_otp_hash, pre_handover_items, contract_review_items, email_verified_override) but they don't exist in prod DB. Features may silently fail.
**Impact**: Security hardening RLS policies NOT active — users may be able to modify `subscription_tier`, `is_admin` directly
**Fix**: Apply all 6 migrations in staging first, test, then prod. This is the #1 risk.

### 🔴 SEC-05: Start-Trial Uses Anon Client to Mutate Billing Fields
**File**: `src/app/api/guardian/start-trial/route.ts:74-81`
**Issue**: Updates `subscription_tier` and `trial_ends_at` using the user's own Supabase session. If schema_v26 RLS is applied (restricts profile self-updates), this route will BREAK. If v26 is NOT applied, users can directly modify these fields via the browser.
**Impact**: Either trial flow breaks, or billing fields are unprotected
**Fix**: Use service-role Supabase client for all entitlement mutations.

### 🔴 SEC-06: Service-Role Falls Back to Anon Key
**Files**: `src/app/api/guardian/phone-verify/route.ts:62-66`, `src/lib/ai/cache.ts:18-24`
**Issue**: `SUPABASE_SERVICE_ROLE_KEY || NEXT_PUBLIC_SUPABASE_ANON_KEY!` — if service role is missing, code silently degrades to unprivileged client, which may succeed doing nothing or fail open.
**Impact**: Silent security degradation
**Fix**: Throw an error if service role key is missing on privileged paths.

### 🔴 SEC-07: Cron Endpoint Uses GET with Secret in Query String
**File**: `src/app/api/cron/cleanup-trials/route.ts:16-24`
**Issue**: `GET /api/cron/cleanup-trials?secret=YOUR_SECRET` — secrets in URL query strings leak into server logs, browser history, referrer headers, monitoring tools.
**Impact**: Secret exposure
**Fix**: Change to POST with `Authorization: Bearer <secret>` header. Return 405 on GET. Rotate `CRON_SECRET`.

### 🔴 SEC-08: Storage RLS May Allow Cross-Tenant Access
**File**: `guide/03-DATA-STORAGE-PLAN.md:96-119`
**Issue**: Storage upload policy checks `bucket_id` and `auth.role() = 'authenticated'` but doesn't enforce user-scoped paths. Any authenticated user can potentially upload to any `projectId/` path.
**Impact**: Cross-tenant file overwrite/access
**Fix**: Use `userId/projectId/...` path pattern and enforce `auth.uid()::text = (storage.foldername(name))[1]` in upload policies.

## P1 — FIX NEXT SPRINT (High)

### 🟡 BUG-03: AI Env Var Name Mismatch
**File**: `src/lib/ai/provider.ts:30`
**Issue**: Code checks `process.env.GOOGLE_AI_API_KEY` but the env example uses `GOOGLE_GENERATIVE_AI_API_KEY`. If the actual env uses the longer name, AI will report "not configured."
**Impact**: AI features randomly disabled
**Fix**: Align env var names across code and .env.

### 🟡 BUG-04: Google OAuth Bypasses Phone Requirement
**File**: `src/app/guardian/login/page.tsx:217-231`
**Issue**: Google OAuth sign-in doesn't collect or require a phone number. The phone verification gate is only on project creation, so OAuth users just skip the signup phone field entirely.
**Impact**: Duplicate account prevention weakened
**Fix**: Enforce phone collection on first dashboard visit post-OAuth.

### 🟡 BUG-05: Open Defect Logic Inconsistent
**Files**: `dashboard/page.tsx:71-76` vs `chat/route.ts:139-145`
**Issue**: Dashboard counts defects NOT in `(verified, rectified)` = 4 statuses. Chat includes only `(open, in_progress)` = 2 statuses. So `reported` and `disputed` defects are "open" on dashboard but invisible to AI chat.
**Impact**: AI gives inaccurate defect counts
**Fix**: Create a shared constant `OPEN_DEFECT_STATUSES = ['open', 'reported', 'in_progress', 'disputed']`.

### 🟡 BUG-06: Free Tier Limits Declared But Not Enforced on Dashboard
**File**: `src/app/guardian/dashboard/page.tsx:39-41`
**Issue**: `FREE_DEFECT_LIMIT` and `FREE_VARIATION_LIMIT` are declared but never used in the dashboard rendering or logic.
**Impact**: Dead code — limits only enforced in project detail components
**Fix**: Remove unused constants or add enforcement.

### 🟡 BUG-07: Multiple In-Progress Stages = Nondeterministic AI Context
**File**: `src/app/api/guardian/ai/chat/route.ts:129-137`
**Issue**: Queries for `status = 'in_progress'` with `limit(1)` — if multiple stages are in_progress, which one is returned depends on DB order (undefined).
**Impact**: AI chat gives advice for wrong stage
**Fix**: Add `order('created_at', { ascending: true })` or use a deterministic stage resolution.

### 🟡 SEC-09: No Next.js Middleware for Route Protection
**Issue**: Every authenticated page checks `getUser()` individually. Missing a check on one page = unauthenticated access.
**Impact**: Auth bypass risk on new pages
**Fix**: Add `middleware.ts` that protects `/guardian/(dashboard|projects|profile|admin|support|refer)`.

### 🟡 SEC-10: Admin Auth Is Hardcoded in Source Code
**File**: `src/lib/admin.ts:2-5`
**Issue**: Admin emails are hardcoded. Adding/removing admins requires a code deploy.
**Impact**: Operational risk, no audit trail
**Fix**: Move to env var `ADMIN_EMAILS=email1,email2` at minimum; ideally use DB `is_admin` column as source of truth.

### 🟡 SEC-11: Admin Actions Use User's Own Supabase Client
**File**: `src/app/guardian/actions.ts:184-190`
**Issue**: `requireAdmin()` creates a client using the user's cookies. If schema_v26 restricts profile updates to service role only, admin grant-trial/revoke/set-tier will fail.
**Impact**: Admin actions broken under hardened RLS
**Fix**: Admin actions should use service-role Supabase client.

### 🟡 SEC-12: No Security Headers in next.config.ts
**File**: `next.config.ts` (empty)
**Issue**: No CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy headers.
**Impact**: XSS/clickjacking/MIME sniffing risks
**Fix**: Add headers configuration.

### 🟡 SEC-13: Client-Side Login Rate Limit Is Bypassable
**File**: `src/lib/security/rate-limit.ts`
**Issue**: Uses `sessionStorage` — clearing session, using incognito, or using curl bypasses it entirely. Server-side AI rate limiter is in-memory (per-instance), useless across serverless cold starts.
**Impact**: Brute force login and AI abuse possible
**Fix**: Add server-side rate limiting (Upstash Redis or Supabase-based). Or use Supabase's built-in auth rate limits.

### 🟡 PERF-01: touchLastSeen Fires on Every Page Render
**File**: `src/app/guardian/actions.ts:8-15`, `dashboard/page.tsx:18-20`
**Issue**: Every dashboard render writes to `profiles.last_seen_at`. This is write amplification.
**Impact**: Unnecessary DB writes, potential hot-row contention
**Fix**: Only update if last_seen_at is >15 minutes old.

### 🟡 PERF-02: Dashboard Makes Serial Queries
**File**: `src/app/guardian/dashboard/page.tsx:22-86`
**Issue**: Auth check → profile fetch → projects fetch → variations fetch → defects fetch → announcements, all sequential.
**Impact**: Slow page load
**Fix**: Use `Promise.all()` for independent queries after auth.

### 🟡 SEO-01: Root Layout SEO Is Tools-Centric, Not Guardian-Centric
**File**: `src/app/layout.tsx:18-55`
**Issue**: Title says "90+ Free Online Tools, Games & HomeOwner Guardian" — Guardian is a footnote. Description says "No sign-up required" which is false for Guardian.
**Impact**: Diluted Guardian SEO
**Fix**: Give Guardian its own metadata in `src/app/guardian/layout.tsx` or `page.tsx`.

### 🟡 SEO-02: AdSense and GA4 IDs Hardcoded
**File**: `src/app/layout.tsx:62-63`
**Issue**: `ADSENSE_PUB_ID` and `GA4_ID` are hardcoded strings, not env vars. Staging/dev will send real analytics data.
**Impact**: Polluted analytics, no staging isolation
**Fix**: Use `process.env.NEXT_PUBLIC_GA_MEASUREMENT_ID` and `process.env.NEXT_PUBLIC_ADSENSE_CLIENT_ID`.

## P2 — IMPROVEMENTS (Medium)

### 🔵 ARCH-01: Guardian Layout Is a Full Client Component
**File**: `src/app/guardian/layout.tsx`
**Issue**: `"use client"` on the layout means the entire Guardian subtree is client-rendered, including server components that could benefit from SSR.
**Impact**: Larger JS bundle, slower initial render
**Fix**: Only wrap the parts that need client behavior (Toast, ErrorBoundary) in client components.

### 🔵 ARCH-02: No Central Error Reporting
**Issue**: Errors are console.error'd and swallowed. No Sentry/LogRocket/similar.
**Impact**: Production bugs go undetected
**Fix**: Add Sentry or similar error tracking.

### 🔵 ARCH-03: deleteProject Is Non-Atomic Sequential Deletes
**File**: `src/app/guardian/actions.ts:49-127`
**Issue**: 14 sequential delete calls. If one fails midway, project is in a partially-deleted state.
**Impact**: Orphaned data, inconsistent state
**Fix**: Use `ON DELETE CASCADE` on all FKs and just delete the project row.

### 🔵 FUNC-01: Referral Reward Is Not Idempotent
**File**: `src/app/api/guardian/referral-reward/route.ts`
**Issue**: No unique constraint preventing the same referred user from earning multiple rewards. Race conditions can cause lost updates on `referral_count`.
**Fix**: Add a `referral_events` table with unique `(referrer_id, referred_id)` constraint.

### 🔵 FUNC-02: Trial + Referral Race Condition
**Files**: `start-trial/route.ts:71-81`, `referral-reward/route.ts:123-138`
**Issue**: Both routes read `trial_ends_at`, calculate a new end date, and write. Concurrent calls can overwrite each other.
**Fix**: Use a DB function with `FOR UPDATE` lock.

### 🔵 FUNC-03: Support Messages Have No Rate Limiting
**File**: `src/app/guardian/actions.ts:392-406`
**Issue**: Users can spam support messages without throttling (only length check).
**Fix**: Add per-user message rate limit (e.g., max 5 messages per minute).

### 🔵 UX-01: Password Toggle Not Keyboard Accessible
**File**: `src/app/guardian/login/page.tsx:362-367`
**Issue**: `tabIndex={-1}` makes the password visibility toggle unreachable by keyboard.
**Fix**: Remove `tabIndex={-1}`.

### 🔵 UX-02: Auth Messages Not Announced to Screen Readers
**File**: `src/app/guardian/login/page.tsx:387-397`
**Issue**: Success/error messages lack `role="alert"` or `aria-live="polite"`.
**Fix**: Add `role="alert"` to the message div.

### 🔵 UX-03: Full Page Reload on Login Success
**File**: `src/app/guardian/login/page.tsx:90`
**Issue**: `window.location.href` causes full page reload instead of client-side navigation.
**Fix**: Use Next.js `router.push()` for smoother UX.

### 🔵 PERF-03: No Image Optimization Config
**File**: `next.config.ts`
**Issue**: No `images.remotePatterns` configured for Supabase Storage URLs. Next.js image optimization won't work for uploaded photos.
**Fix**: Add Supabase Storage domain to `images.remotePatterns`.
